# Functional Requirements Specification

## Feature Name

Workspace Initialization

## Description

Creates or refreshes a local `.l10n-audit/` workspace and generates configuration based on project detection.

## Inputs

- Project path.
- Optional channel.
- Optional force flag.

## Outputs

- `.l10n-audit/` workspace.
- Configuration files and detected project metadata.

## Business Logic

- Detect supported project profile and locale layout.
- Create isolated working directories for audit execution.
- Avoid overwriting existing configuration unless force is requested.

## Validation Rules

- Project path must exist.
- Existing workspace overwrite must be explicit.
- Generated configuration must remain schema-compatible.

## Error Handling

- Invalid project paths return project validation errors.
- Existing workspace conflicts require force or user action.

## User Flow

1. User runs `l10n-audit init`.
2. Toolkit detects project layout.
3. Toolkit creates workspace and configuration.
4. User proceeds to audit execution.

## Acceptance Criteria

- A valid workspace exists after successful initialization.
- Detected profile and locale paths are visible to the user.
- Source files are not modified.

## Feature Name

Audit Execution Pipeline

## Description

Runs deterministic and optional AI-assisted localization audits against isolated locale files.

## Inputs

- Project path.
- Audit stage: `fast`, `full`, `grammar`, `terminology`, `placeholders`, `ar-qc`, `ar-semantic`, `icu`, `reports`, `autofix`, `ai-review`, or `camel`.
- Configuration values.
- Optional glossary.
- Optional AI provider settings.

## Outputs

- Normalized findings.
- `Results/artifacts/audit_master.json`.
- `Results/final/final_audit_report.json` and Markdown report.
- `Results/review/review_queue.xlsx`.
- Optional per-tool artifacts.

## Business Logic

- Prepare an isolated runtime.
- Execute the selected stage modules.
- Normalize findings into stable issue codes and report structures.
- Aggregate results into master state.
- Project reviewable findings into the review queue.

## Validation Rules

- Stage must be one of the supported stage names.
- AI configuration must be valid when AI review is enabled.
- Runtime configuration must be loadable and schema-compatible.

## Error Handling

- Invalid stage fails before execution.
- Missing AI credentials fail AI-enabled requests.
- Module failures are reported through structured audit errors.

## User Flow

1. User runs `l10n-audit run --stage fast` or another supported stage.
2. Toolkit isolates locale inputs.
3. Toolkit runs configured modules.
4. Toolkit generates reports and review artifacts.

## Acceptance Criteria

- Selected stages generate expected artifacts.
- Deterministic stages do not require AI.
- Source files are not modified during audit.

## Feature Name

Human Review Queue

## Description

Projects review-required and actionable findings into an editable workbook for human decisions.

## Inputs

- Aggregated findings.
- Master artifact.
- Current locale values.

## Outputs

- `Results/review/review_queue.xlsx`.

## Business Logic

- Include actionable and review-required rows.
- Suppress non-actionable informational rows where required by pipeline rules.
- Preserve issue identity, source context, target value, suggestion, status, and review notes.

## Validation Rules

- Review queue schema must remain stable.
- Each row must preserve enough identity for freezing and apply validation.

## Error Handling

- Missing source values are represented safely and must not be treated as valid apply inputs.
- Invalid projections fail report generation rather than producing unsafe contracts.

## User Flow

1. User opens `review_queue.xlsx`.
2. User approves, rejects, or edits suggested changes.
3. User runs `prepare-apply`.

## Acceptance Criteria

- Reviewers can make decisions without editing source locale files.
- Review data can be frozen into `review_final.xlsx`.

## Feature Name

Prepare Apply

## Description

Freezes approved rows from the editable review queue into the execution-only `review_final.xlsx` workbook.

## Inputs

- `review_queue.xlsx`.
- Optional output path for `review_final.xlsx`.
- Optional rejection report path.

## Outputs

- `Results/review/review_final.xlsx`.
- Optional rejection report.

## Business Logic

- Validate approved rows.
- Reject incomplete, stale, unresolved, or unsafe rows.
- Generate a deterministic apply contract.

## Validation Rules

- Approved rows must have valid source identity and approved target values.
- Frozen workbook schema must match the apply contract.
- `review_final.xlsx` must be generated by the tool, not manually edited.

## Error Handling

- Invalid rows are rejected and reported.
- Contract generation fails closed on schema violations.

## User Flow

1. User finishes human review.
2. User runs `l10n-audit prepare-apply`.
3. Toolkit creates `review_final.xlsx`.
4. User runs `l10n-audit apply`.

## Acceptance Criteria

- Only approved, valid rows enter the frozen workbook.
- Rejected rows are explainable.

## Feature Name

Apply Approved Fixes

## Description

Applies approved changes from the frozen workbook to generated locale fix outputs.

## Inputs

- `review_final.xlsx`.
- Project path.
- Runtime configuration.

## Outputs

- Framework-compatible `.fix` locale files.
- Reconciliation updates in master artifacts.

## Business Logic

- Read only the frozen execution contract.
- Validate source hashes and approved target values.
- Write exact-format fix files through locale exporters.
- Reconcile apply results into master state.

## Validation Rules

- `review_queue.xlsx` must not be used as apply input.
- Source hash mismatch must block the row.
- Tampered approved values must be rejected.
- Unresolved lookup sentinels must not be treated as valid source identity.

## Error Handling

- Stale, tampered, or unresolved rows fail closed.
- Unsupported locale formats produce explicit errors.

## User Flow

1. User runs `l10n-audit apply`.
2. Toolkit reads `review_final.xlsx`.
3. Toolkit validates each approved row.
4. Toolkit writes fix outputs and reconciles state.

## Acceptance Criteria

- No unapproved row is applied.
- Apply writes only expected fix outputs.
- Reconciliation records the apply outcome.

## Feature Name

Adaptive Configuration Workflow

## Description

Transforms learning-profile data into reviewed configuration changes through an explicit manifest workflow.

## Inputs

- Learning profile JSON.
- Adaptation report path.
- Manifest path.
- Approvals JSON.
- Reviewed manifest path.
- Optional target config path.

## Outputs

- `adaptation_report.json`.
- Consumption manifest.
- Reviewed manifest.
- Updated `config.json` after approved apply.

## Business Logic

- Generate adaptation signals without automatically changing runtime behavior.
- Convert allowed signals into bounded manifest actions.
- Require explicit approvals.
- Apply only approved reviewed manifest actions.

## Validation Rules

- Mode must be explicit.
- Allowed signal keys and action types must be configured.
- Manifest and reviewed manifest must match expected contracts.

## Error Handling

- Invalid or missing inputs fail before config mutation.
- Unapproved actions are ignored.

## User Flow

1. User runs `generate-adaptation-report`.
2. User runs `generate-manifest`.
3. User supplies approvals and runs `review-manifest`.
4. User runs `apply-manifest`.

## Acceptance Criteria

- No configuration change occurs without reviewed approval.
- Applied changes are deterministic and traceable.

## Feature Name

Optional AI Review

## Description

Uses an OpenAI-compatible provider through LiteLLM to propose translation improvements and semantic review suggestions.

## Inputs

- AI enabled flag.
- Provider, model, API base, and API key.
- Source and target locale strings.
- Audit findings and context.

## Outputs

- AI suggestions.
- Review-only findings.
- AI status in CLI/report output.

## Business Logic

- AI review augments deterministic checks.
- Suggestions must pass validation before entering review artifacts.
- Provider calls use bounded retries and rate-limit-aware behavior.

## Validation Rules

- AI must not bypass source identity or apply-contract validation.
- Missing API key fails AI-enabled execution.
- Short strings and domain-sensitive labels require conservative handling.

## Error Handling

- Provider failures are surfaced without corrupting deterministic artifacts.
- Debug output is controlled by configured flags.

## User Flow

1. User enables AI review.
2. Toolkit validates provider configuration.
3. Toolkit requests suggestions.
4. Human reviewer approves or rejects suggested changes.

## Acceptance Criteria

- AI output never applies directly without review/freeze/apply.
- AI status is visible to the user.

## Feature Name

Workspace Doctor

## Description

Inspects project and workspace discovery health.

## Inputs

- Project path.
- Optional AI check flag.

## Outputs

- Structured health report containing success status, profile, translation paths, warnings, and errors.

## Business Logic

- Validate project layout.
- Inspect configured locale paths.
- Optionally verify AI readiness.

## Validation Rules

- Project path must exist.
- Output schema must remain stable for API consumers.

## Error Handling

- Invalid paths and configuration problems return explicit warnings or errors.

## User Flow

1. User runs `l10n-audit doctor`.
2. Toolkit inspects configuration and workspace.
3. Toolkit reports actionable status.

## Acceptance Criteria

- Doctor output explains readiness or blockers.
- It does not modify project files.

## Feature Name

HTTP Reference API

## Description

Exposes selected public Python API functions through optional FastAPI endpoints.

## Inputs

- HTTP requests for health, audit run, workspace doctor, and workspace init.

## Outputs

- JSON responses and HTTP errors.

## Business Logic

- Wrap public Python API calls.
- Run heavy audit work in a threadpool.
- Return stable JSON responses.

## Validation Rules

- Request models must include required project paths.
- Invalid projects and AI configuration errors must map to appropriate HTTP errors.

## Error Handling

- Invalid project: HTTP 400.
- AI configuration problem: HTTP 422.
- Audit errors: HTTP 500.

## User Flow

1. User starts the FastAPI server.
2. User calls `/health`, `/workspace/init`, `/workspace/doctor`, or `/audit/run`.
3. API returns JSON output.

## Acceptance Criteria

- HTTP API behavior mirrors public Python API behavior.
- CLI remains the primary supported interface.
