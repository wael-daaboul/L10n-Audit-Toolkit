# Database Documentation

## Overview

L10n Audit Toolkit does not use a relational database, document database, or remote persistence service. The system uses file-based data artifacts as its persistent data model.

These artifacts must be treated as data stores with stable contracts because they drive review, apply, reporting, and traceability.

## Tables / Collections

### `audit_master.json`

Central master state for audit findings, projection, reconciliation, and provenance.

### `review_queue.xlsx`

Editable workbook used by humans to review suggestions and record approval decisions.

### `review_final.xlsx`

Frozen execution contract generated by `prepare-apply` and consumed by `apply`.

### `final_audit_report.json`

Machine-readable final report for downstream automation.

### `final_audit_report.md`

Human-readable dashboard report.

### `fix_plan.json`

Safe-fix plan artifact when deterministic safe-fix logic is used.

### Manifest Artifacts

Adaptive workflow JSON artifacts:

- `adaptation_report.json`
- consumption manifest
- reviewed manifest
- application receipts

### `.fix` Locale Files

Generated framework-compatible fixed locale output files.

## Fields

### Common Finding Fields

- `key`: translation key.
- `locale`: affected locale.
- `issue_type`: legacy issue classification.
- `code`: normalized machine-readable issue code.
- `severity`: `error`, `warning`, or `info`.
- `message`: human-readable issue description.
- `source`: source locale value where available.
- `target`: current target locale value where available.
- `suggestion` or `suggested_fix`: proposed correction.
- `approved_new`: approved target value used only by apply-layer contracts.
- `needs_review`: whether human review is required.
- `extra`: module-specific metadata.

### Review Workbook Fields

The exact workbook schemas are safety contracts protected by tests. Conceptual fields include:

- stable plan or row identity.
- translation key.
- locale.
- source text.
- current target text.
- issue category.
- suggested correction.
- reviewer status.
- approved final value.
- notes and trace metadata.
- source hash or identity metadata.

### Manifest Fields

Conceptual manifest fields include:

- signal key.
- action type.
- target configuration path.
- proposed value.
- approval status.
- review metadata.
- application receipt data.

## Relationships

- `audit_master.json` is the central state that feeds reports and review projection.
- `review_queue.xlsx` is derived from audit findings and current values.
- `review_final.xlsx` is derived from approved, valid rows in `review_queue.xlsx`.
- `.fix` locale files are derived from `review_final.xlsx`.
- Reconciliation writes apply outcomes back into the master state.
- Adaptive manifests are derived from learning profiles and adaptation reports, then applied to `config.json` only after approval.

## Indexes

No database indexes exist. Logical lookup keys include:

- translation key.
- locale.
- issue code.
- plan identity.
- source hash.
- artifact path.

These logical keys are used for deduplication, validation, reconciliation, and traceability.

## Constraints

- `review_final.xlsx` must be generated by the toolkit.
- Apply must not consume `review_queue.xlsx`.
- Source identity must match before a row is applied.
- Unresolved lookup sentinels are invalid apply sources.
- `approved_new` must not be inferred from suggestions during apply validation.
- Workbook schemas must remain stable unless migration notes and tests are updated.
- Generated artifacts must not store AI credentials.

## Validation Rules

- Configuration must conform to supported schema rules.
- Audit stages must emit findings that can normalize into known issue codes or explicit unknown classifications.
- Report aggregation must preserve provenance.
- Prepare-apply must reject unsafe rows.
- Apply must reject stale, tampered, unresolved, or incomplete rows.
- Manifest apply must reject missing or unapproved actions.

## CRUD Operations

| Artifact | Create | Read | Update | Delete |
| :--- | :--- | :--- | :--- | :--- |
| `.l10n-audit/` workspace | `init` | `run`, `doctor` | `update`, `init --force` | manual or reset workflows |
| `audit_master.json` | `run` / aggregation | reports, apply, reconciliation | reconciliation | result cleanup/reset |
| `review_queue.xlsx` | report aggregation | reviewer, `prepare-apply` | human reviewer | result cleanup/reset |
| `review_final.xlsx` | `prepare-apply` | `apply` | regenerate only | result cleanup/reset |
| final reports | report aggregation | user, CI/CD | regenerate | result cleanup/reset |
| manifest artifacts | adaptive commands | reviewer, manifest apply | reviewed-manifest generation | result cleanup/reset |
| `.fix` files | `apply` | developer/CI | regenerate | manual cleanup |

## Security Considerations

- Review and final workbooks should be treated as controlled release artifacts.
- Storing generated artifacts in shared environments requires normal repository and filesystem access controls.
- API keys must remain in environment variables or secure CI/CD secret stores.
- The optional HTTP API has no built-in authentication and should not be exposed publicly without additional controls.
- Generated fix outputs should be reviewed before merging into production branches.
