# Architecture Decision Records

## ADR-001: CLI-First Architecture

## Context

The toolkit must work for developers, QA teams, localization reviewers, and CI/CD systems without requiring hosted infrastructure.

## Decision

Use a CLI-first architecture with `l10n-audit` as the primary interface and expose public Python functions for integration.

## Alternatives Considered

- Hosted web application.
- Desktop GUI application.
- API-only service.

## Consequences

- Users can run audits locally and in CI/CD.
- Installation and environment setup are the main onboarding requirements.
- UI is intentionally artifact-driven through CLI output, Markdown reports, and Excel workbooks.

## Reasoning

Localization QA must fit existing developer pipelines. A CLI keeps the tool scriptable, portable, and independent from server availability.

## ADR-002: File-Based Artifact State Instead of Server Database

## Context

The pipeline needs traceability, review workbooks, deterministic reports, and apply contracts, but it does not require multi-user transactional storage.

## Decision

Use JSON and XLSX artifacts under `Results/` and `.l10n-audit/` instead of a database server.

## Alternatives Considered

- SQLite database.
- PostgreSQL or another server database.
- Stateless report-only output.

## Consequences

- Artifacts are easy to archive and inspect.
- CI/CD integration remains simple.
- Concurrent multi-user editing is not handled by the toolkit.
- Artifact schemas become important public contracts.

## Reasoning

The product value is deterministic audit/review/apply flow, not centralized data hosting. File artifacts are sufficient and operationally simpler for the current scope.

## ADR-003: Isolated Audit Workspace

## Context

Audit modules inspect and normalize locale content. Direct source mutation during audit would create release risk.

## Decision

Run audits against an isolated `.l10n-audit/` workspace and keep source mutation out of audit stages.

## Alternatives Considered

- Run directly against project source files.
- Apply safe fixes during audit by default.

## Consequences

- Users can run audits without fear of source changes.
- Apply becomes a separate, explicit workflow.
- Runtime setup requires workspace preparation.

## Reasoning

Safety and trust are more important than minimizing pipeline steps. Isolation keeps audit behavior predictable.

## ADR-004: Human Review Queue and Frozen Apply Contract

## Context

Translation fixes can be linguistically sensitive. Editable review artifacts are useful, but editable artifacts are unsafe as direct apply inputs.

## Decision

Separate `review_queue.xlsx` from `review_final.xlsx`. The queue is human-editable; the final workbook is generated by `prepare-apply` and consumed by `apply`.

## Alternatives Considered

- Apply directly from `review_queue.xlsx`.
- Store approvals only in JSON.
- Auto-apply AI and deterministic suggestions in one step.

## Consequences

- Workflow has an extra freeze step.
- Apply safety is stronger and easier to audit.
- Workbook schemas must remain stable and tested.

## Reasoning

A frozen execution contract prevents accidental edits, stale decisions, and unsafe direct application of review workspace data.

## ADR-005: Source Identity Validation Before Apply

## Context

Locale files may change between audit, review, and apply. Applying stale changes can corrupt translations.

## Decision

Validate source identity and approved target values before applying each row.

## Alternatives Considered

- Trust review workbook values.
- Re-run audit automatically during apply.
- Prompt user for conflict decisions interactively.

## Consequences

- Stale or tampered rows fail closed.
- Apply is safer and deterministic.
- Users may need to rerun audit/review when source files change.

## Reasoning

Fail-closed behavior is required for production-safe localization changes.

## ADR-006: Optional AI as Review Augmentation

## Context

AI can improve semantic review coverage but provider output is probabilistic and can be wrong.

## Decision

Treat AI review as optional augmentation. AI suggestions enter the review workflow and never bypass frozen apply validation.

## Alternatives Considered

- AI disabled permanently.
- AI auto-applies high-confidence changes.
- AI as mandatory pipeline stage.

## Consequences

- Deterministic stages remain available without AI.
- Reviewers can benefit from AI suggestions while retaining control.
- Provider configuration and resilience must be documented and validated.

## Reasoning

AI is valuable for candidate generation, not as an unchecked authority for applying localization changes.

## ADR-007: Explicit Adaptive Configuration Manifests

## Context

Learning-based configuration changes can improve future audits but automatic configuration mutation creates operational risk.

## Decision

Use a multi-step adaptive workflow: generate adaptation report, generate manifest, review manifest, apply approved manifest.

## Alternatives Considered

- Automatically update `config.json`.
- Emit suggestions only in logs.
- Disable adaptive intelligence.

## Consequences

- Configuration changes are slower but reviewable.
- Project owners control which actions are allowed.
- Manifest schemas and receipts become important artifacts.

## Reasoning

Configuration affects audit behavior and must be governed by explicit human approval.

## ADR-008: Optional HTTP API as Reference Wrapper

## Context

Some users may want an HTTP integration, but the core product should remain CLI-first.

## Decision

Provide a FastAPI reference server that wraps public Python API functions.

## Alternatives Considered

- No HTTP API.
- Make HTTP API the primary product interface.
- Build a full authenticated web service.

## Consequences

- API integrations are possible with low implementation cost.
- Authentication and deployment hardening are out of scope for the reference server.
- CLI and Python API remain the authoritative operational path.

## Reasoning

The HTTP API demonstrates integration potential without forcing server operations onto CLI users.
