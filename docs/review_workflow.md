# Review Workflow

## Purpose

L10n Audit Toolkit separates deterministic fixes from human-reviewed fixes. This document explains the fix plan, review queue, approved fixes, and final locale outputs.

## Workflow Summary

1. Run audits.
2. Aggregate findings into final reports.
3. Generate safe fix candidates.
4. Review risky changes in the XLSX review queue.
5. Apply approved fixes with integrity checks.
6. export the final locale.

## Safe Fixes

Safe fixes are generated by:

- `python -m fixes.apply_safe_fixes`
- or `./bin/run_all_audits.sh --stage autofix`

Outputs:

- `Results/fixes/fix_plan.json`
- `Results/fixes/fix_plan.xlsx`
- `Results/fixes/safe_fixes_applied_report.json`

The fix plan contains candidate changes classified as either:

- `auto_safe`
- `review_required`

Only conservative changes are auto-applied.

## Review Queue

The review queue is written to:

- `Results/review/review_queue.xlsx`

This file is intended for human review of changes that are not safe to apply automatically.

Important fields:

- `old_value`: current locale value at queue-generation time
- `suggested_fix`: machine-generated suggestion
- `approved_new`: reviewer-approved replacement
- `status`: reviewer decision
- `source_hash`: integrity snapshot of the original locale value
- `suggested_hash`: integrity snapshot of the approved content
- `plan_id`: row identity for the review item
- `generated_at`: queue-generation timestamp

## Approved Fixes

To approve a row:

1. confirm the current locale value and suggestion
2. enter the reviewed text in `approved_new`
3. set `status` to `approved`

Apply approved rows with:

```bash
python -m fixes.apply_review_fixes
```

## Integrity Checks

Approved rows are not applied blindly.

The toolkit validates:

- required review columns are present
- duplicate approved rows for the same key and locale are rejected
- conflicting approved rows for the same key and locale are rejected
- the current locale value still matches the recorded source snapshot
- the approved value still matches the recorded suggestion hash

If an integrity check fails, the row is skipped and the reason is recorded in the final report.

## Final Outputs

After reviewed fixes are applied, the toolkit writes:

- `Results/final_locale/ar.final.json`
- `Results/final_locale/review_fixes_report.json`

The report summarizes:

- applied approved rows
- skipped rows
- stale rows
- conflicting rows
- malformed rows

## Recommended Review Discipline

- review high-severity placeholder and ICU findings first
- treat terminology and context-sensitive findings as human-decision items
- do not edit integrity fields manually
- regenerate the review queue if source locale files changed after review started
